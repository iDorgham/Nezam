# Swarm workflow lifecycle

This document is the canonical operating workflow for the COIA 13-swarm agent system. It defines how the Chief Project Orchestrator, Deputy Orchestrator, Swarm Managers, Team Managers, and Specialists collaborate from project intake through maintenance.

Related governance:
- Agent hierarchy and catalog: [`.cursor/agents/README.md`](../../../../.cursor/agents/README.md)
- Runtime coordinator: [`.cursor/agents/orchestration-subagent-controller.md`](../../../../.cursor/agents/orchestration-subagent-controller.md)
- Prompt aliases and routing matrix: [`ORCHESTRATION_ALIASES.md`](ORCHESTRATION_ALIASES.md)

## Operating hierarchy

```text
Tier 0  CPO ............................... final scope, strategy, go/no-go
Tier 0  Deputy Orchestrator ............... daily cadence, conflicts, gate tracking
Tier 1  Swarm Manager ..................... domain ownership across 13 swarms
Tier 2  Team Manager ...................... subdomain planning and handoff quality
Tier 3  Specialist ........................ concrete analysis, implementation, review
```

Escalation chain:
1. Specialist -> Team Manager
2. Team Manager -> Swarm Manager
3. Swarm Manager -> Deputy Orchestrator
4. Deputy Orchestrator -> CPO

Ethics veto path:
- `lead-ai-ethics-officer` may escalate unresolved critical ethical risks directly to `cpo` for veto/no-go decisions.

## Global rules

- Preserve SDD order: Planning -> SEO/IA -> Content -> Design -> Development -> Hardening -> Release.
- Development remains hardlocked until required PRD, project prompt, design contract, gate matrix, and active prompt artifacts exist.
- Every routed task must name the Swarm Manager, Team Manager, Specialists, allowed paths, acceptance checks, and validation evidence.
- Parallel swarm work is allowed only when write scopes are independent or the Deputy Orchestrator owns merge coordination.
- Cross-swarm conflict resolution prioritizes quality first, speed second, and cost third.
- Decisions that change scope, contracts, gates, risk posture, or architecture must be persisted in project docs before execution continues.
- Reports generated by scans, audits, or verification loops must be written under `docs/reports/<category>/`.

## Phase 1: Intake and initial assessment

Ethics gate (Phase 1):
- Classify AI/non-AI scope and trigger `lead-ai-ethics-officer` for user-facing AI, sensitive data, or model-risk features.

Owner: CPO  
Coordinator: Deputy Orchestrator  
Primary swarms: Architecture & Planning, Security, UI/UX Design, CMS & SaaS, Analytics & Dashboard as needed

### Entry criteria

- Project request, client brief, issue, or product goal is available.
- Known constraints, deadlines, target users, and success metrics are captured or explicitly marked unknown.
- Repository and governed docs are available for inspection when implementation may follow.

### Activities

1. CPO classifies the project type, risk level, target outcome, and impacted swarms.
2. Deputy Orchestrator prepares the initial routing packet.
3. Requirements Analysis Manager identifies missing inputs and acceptance criteria.
4. Security, compliance, data, mobile, payment, localization, and CMS specialists are pulled in only when their risk domain is present.
5. CPO decides whether the work can proceed, requires more discovery, or must be blocked.

### Exit evidence

- Intake summary with objective, constraints, assumptions, risks, and impacted swarms.
- Initial owner map: CPO -> Deputy -> Swarm Manager(s) -> Team Manager(s).
- Next legal command, usually `/PLAN`, `/GUIDE`, or a narrow `/SCAN`.

## Phase 2: Planning, architecture, SEO/IA, content, and design

Ethics gate (Phase 2):
- Complete ethics design review (fairness, privacy, transparency, safety, IP, sustainability) before architecture sign-off.

Owner: CPO  
Coordinator: Deputy Orchestrator  
Primary swarms: Architecture & Planning, UI/UX Design, Frontend, Backend, Data & Database, CMS & SaaS, Security

### Entry criteria

- Intake evidence exists.
- Required PRD and project prompt status is known.
- Active plan location is known under `docs/workspace/plans/**` or `.ai/plans/**`.

### Activities

1. Requirements Analysis Manager converts goals into specs, acceptance criteria, and dependency maps.
2. Solution Design Manager creates architecture options, ADRs, and trade-off notes.
3. Integration Architecture Manager maps external systems, auth, payment, CMS, analytics, and data flows.
4. SEO, IA, content, and localization lenses define route intent, content scope, entity structure, and language requirements where relevant.
5. UI/UX Design swarm defines flows, accessibility expectations, token changes, component contracts, and `docs/DESIGN.md`.
6. Security swarm performs threat-model and compliance planning for high-risk areas.
7. Deputy Orchestrator verifies that active phase prompt artifacts exist before development routing.

### Exit evidence

- Approved PRD/specs, acceptance criteria, and risk register.
- Architecture decision record or equivalent design summary.
- `docs/DESIGN.md` updated for UI-facing scope.
- Active phase `prompt.json` and `PROMPT.md`.
- Handoff packets for each downstream swarm.

## Phase 3: Sprint development and swarm execution

Ethics gate (Phase 3):
- Implement guardrails, logging, and policy controls required by ethics review.

Owner: Relevant Swarm Manager  
Coordinator: Deputy Orchestrator  
Primary swarms: Frontend, Backend, Data & Database, Mobile, CMS & SaaS, Analytics & Dashboard, Performance & DevOps

### Entry criteria

- Phase 2 exit evidence is complete.
- Allowed paths and write ownership are explicit.
- Acceptance checks and validation commands are assigned.
- Parallel work has non-overlapping scope or explicit Deputy coordination.

### Activities

1. Swarm Manager decomposes work into team-level tasks.
2. Team Managers assign Specialists and define concrete deliverables.
3. Specialists implement or produce focused artifacts inside allowed paths.
4. Code Generation Supervisor reviews agent-generated code for scope, slop, and policy drift when AI-assisted output is material.
5. Daily Sync Agent gathers progress, blockers, risks, and validation status.
6. Knowledge Sharing Agent records durable decisions and handoff updates.

### Exit evidence

- Implemented change or completed artifact per task.
- Task-level validation evidence.
- Updated docs when contracts, behavior, or architecture changed.
- No unresolved owner conflicts or shared-scope collisions.

## Phase 4: Integration, testing, and hardening

Ethics gate (Phase 4):
- Run fairness and misuse red-team checks; unresolved critical findings block release.

Owner: Lead QA Architect  
Coordinator: Deputy Orchestrator  
Primary swarms: Quality Assurance, Security, Performance & DevOps, Frontend, Backend, Data & Database, Mobile

### Entry criteria

- Development tasks report complete with validation evidence.
- Integration environments, fixtures, and test data are available or gaps are documented.
- Risk areas from planning are mapped to test coverage.

### Activities

1. Testing Manager runs functional, integration, regression, and E2E checks appropriate to the scope.
2. Automation Manager verifies repeatable test automation and CI readiness.
3. Performance Load Manager checks load, latency, and Core Web Vitals budgets where relevant.
4. Security swarm runs threat, dependency, authz, input, and exposure checks.
5. Data & Database swarm verifies migrations, rollback, data integrity, and observability queries.
6. Conflict Resolution Agent arbitrates fixes that cross domains.

### Exit evidence

- Test matrix with pass/fail status and known gaps.
- Security findings are fixed or recorded as accepted risks.
- Performance/a11y budget status is clear.
- Release blocker list is empty or explicitly deferred by CPO.

## Phase 5: Deployment, launch, and release governance

Ethics gate (Phase 5):
- Obtain launch ethics sign-off for high-risk AI/data scope; escalate veto decisions to `cpo`.

Owner: Lead DevOps & Performance  
Coordinator: CPO and Deputy Orchestrator  
Primary swarms: Performance & DevOps, Security, Quality Assurance, Analytics & Dashboard, Maintenance & Support

### Entry criteria

- Phase 4 exit evidence is complete.
- Rollback plan, environment config, secrets posture, and monitoring are verified.
- Release notes, migration notes, and stakeholder communication are ready when applicable.

### Activities

1. DevOps Manager verifies CI/CD, environments, secrets, migrations, and deployment sequence.
2. Infrastructure Manager checks capacity, regions, scaling, and rollback levers.
3. Observability Specialist confirms dashboards, traces, logs, alerts, and SLO signals.
4. Security and compliance owners verify release posture.
5. Analytics & Dashboard swarm verifies launch metrics and event tracking.
6. CPO gives final go/no-go.

### Exit evidence

- Deployment record or release checklist.
- Monitoring links, rollback instructions, and ownership rotation.
- Launch metrics baseline.
- Post-launch watch window and escalation plan.

## Phase 6: Maintenance, support, and knowledge compounding

Ethics gate (Phase 6):
- Monitor drift, abuse, and ethics regressions; record updates via `knowledge-sharing-agent`.

Owner: Lead Maintenance Agent  
Coordinator: Deputy Orchestrator  
Primary swarms: Maintenance & Support, Quality Assurance, Security, Performance & DevOps, Analytics & Dashboard

### Entry criteria

- Product, feature, or platform surface is live or in sustained use.
- Incoming bugs, support issues, incidents, drift signals, or dependency updates exist.

### Activities

1. Bug Triage Manager classifies severity, owner, repro evidence, and duplicate status.
2. Tech Debt Manager maintains refactor and reliability backlog.
3. Knowledge Update Manager updates durable context and docs after meaningful changes.
4. Analytics swarm monitors adoption, errors, funnels, and support signals.
5. Security and DevOps swarms handle patches, incidents, rotations, and dependency risk.
6. CPO reviews recurring signals for roadmap or strategy impact.

### Exit evidence

- Triage digest, incident note, or debt backlog update.
- Documentation and memory updates for durable learnings.
- Follow-up plan or next legal command.

## Swarm Communication Protocol

### Handoff Packet (required between every phase)

Every agent that completes a phase must emit a handoff packet before the next agent begins:

```yaml
handoff:
  from_agent: [agent name]
  to_agent: [next agent name]
  phase_completed: [phase ID]
  phase_started: [next phase ID]
  artifacts_produced:
    - path: [file path]
      status: approved | draft | pending-review
  gates_passed:
    - gate: [gate name]
      evidence: [brief description]
  open_items:
    - [Any unresolved items the next agent must know about]
  plain_language_summary:
    what_happened: [1 sentence]
    what_the_next_agent_needs_to_do: [1 sentence]
  escalation_required: false | true
  escalation_reason: [if true, why]
```

### Design -> Content Communication Contract

Design lead MUST emit before content work begins:
- Approved DESIGN.md with typography scale
- Component vocabulary (names and variants) for copy consistency
- Voice/tone guidance integration points (where microcopy appears in components)
- RTL readiness confirmation (if applicable)

Content team MUST emit before development begins:
- CONTENT_MAP.md with all page copy finalized
- SEO meta titles and descriptions per page
- All microcopy strings (button labels, error messages, empty states, loading states)
- Structured data schema per page type

### Content -> Development Communication Contract

Frontend lead receives from content:
- All copy strings in structured format (page -> section -> element)
- Character/word limits per UI context
- Localization requirements (if multilingual)

### Design -> Development Communication Contract

Frontend lead receives from design:
- DESIGN.md (authoritative)
- Component API specs with prop types and variants
- Motion tokens (duration, easing, reduced-motion alternatives)
- Responsive breakpoint matrix
- Token references for all style decisions

### Escalation Paths

| Situation | Escalate to |
|---|---|
| Gate blocked, no clear path forward | swarm-leader.md |
| Architecture conflict between agents | project-architect.md |
| Design token violation in implementation | design-lead.md |
| Security issue found during development | security-auditor.md |
| Performance budget exceeded | performance-engineer.md |
| Accessibility violation discovered | a11y-performance-auditor.md |
| Content-design mismatch | design-lead.md + content-strategist.md |

### Daily Sync Format

Each active agent emits a brief status at end of session:

```yaml
daily_sync:
  agent: [name]
  date: [YYYY-MM-DD]
  completed_today:
    - [task or artifact]
  in_progress:
    - [task]
  blocked_by:
    - [blocker] | none
  tomorrow_priority:
    - [task]
```

## Cadence

- Daily: Deputy Orchestrator runs cross-swarm status through Daily Sync Agent.
- Per task: Team Manager verifies handoff packet quality before Specialist work begins.
- Per phase: CPO or delegated Swarm Manager verifies entry and exit gates.
- Per release: DevOps, Security, QA, and Analytics provide go/no-go evidence.
- Post-release: Maintenance swarm compounds learnings and routes recurring issues back into planning.

## Required handoff packet

Every delegated task must include:

- Objective
- Phase and swarm
- Owner chain
- Allowed paths
- Inputs and source artifacts
- Dependencies and blockers
- Acceptance checks
- Validation command or verification method
- Expected output artifact
- Escalation path

## Decision and memory policy

Persist decisions when they affect:

- Architecture or integration contracts
- Security, privacy, compliance, or payments posture
- Data schemas, migrations, indexing, or retention
- Design tokens, component APIs, accessibility behavior, or motion contracts
- Deployment topology, monitoring, rollback, or SLOs
- User-facing copy, localization, or content models

Use the most specific governed location available. Do not write generated reports outside `docs/reports/**`; do not edit generated `.claude/**`, `.antigravity/**`, `CLAUDE.md`, or `AGENTS.md` by hand.

## Go/no-go rubric

Use this rubric before moving between phases:

- Scope is clear and accepted.
- Owner chain is explicit.
- Required upstream artifacts exist.
- Risks are documented and assigned.
- Implementation or artifact evidence exists.
- Validation command or equivalent evidence is recorded.
- Downstream handoff is ready.
- Next legal command is clear.

If any item is false, the decision is `no-go` or `replan`.
